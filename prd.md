Ниже — продуктовые требования (PRD-уровень) для веб-приложения “Годовые цели → месяцы → недели → дни” с фокусом на сверхбыстрый ввод (1–2 клика), визуализацию и трекинг.

---

## 1) Цель продукта и ценность

**Проблема:** людям сложно доводить годовые цели до результата из-за разрыва между “стратегией” и “ежедневными действиями”, перегруженности планированием и отсутствия прозрачной картины прогресса.

**Ценность продукта:** быстро сформулировать годовые цели и автоматически/полуавтоматически разложить их на месяцы/недели/дни, чтобы пользователь ежедневно видел “что делать сегодня” и понимал, как это влияет на годовой результат.

**Ключевой принцип:** минимальный friction — создание/изменение плана должно занимать секунды, а не “сеанс планирования”.

---

## 2) Персоны и основные JTBD

### Персона A: “Занятый профессионал”

* Хочет 3–7 целей на год, без сложных методологий.
* Нужны быстрые ежедневные задачи и ясный прогресс.

### Персона B: “Системный планировщик”

* Любит структуру, повторяющиеся привычки, недельные обзоры.
* Нужны шаблоны, ритуалы планирования, гибкая корректировка.

## 3) Объекты данных (доменные сущности)

1. **Goal (Цель)**

   * id, title, description (опционально), category/area (опционально)
   * time_horizon: year (по умолчанию), start_date, end_date
   * success_metric (опционально): число/валюта/шт/время/булево
   * target_value, current_value
   * status: active/paused/done/archived
   * priority (1–5), confidence (опционально)
   * visibility (личное)

2. **PlanItem (Элемент плана)**

   * type: milestone / task / habit / checkpoint
   * связка: goal_id (обяз.), parent_id (для декомпозиции)
   * period: month/week/day (или конкретные даты)
   * schedule:

     * one-off (дата/диапазон)
     * recurring (RRULE-подобная модель: daily/weekly/monthly + параметры)
   * effort (опционально): minutes / points
   * status: todo/done/skipped/failed
   * completion_date, notes

3. **Review (Обзор)**

   * weekly_review, monthly_review: шаблон вопросов + фиксация решений
   * результаты: что перенесено/удалено/переприоритезировано

4. **Template (Шаблон)**

   * шаблоны целей и декомпозиции (например “выучить язык”, “спорт”)

---

## 4) Информационная архитектура (экраны)

1. **Home / Today**

   * список задач на сегодня (включая повторяющиеся)
   * быстрый ввод (командная строка / кнопка “+”)
   * переключатель “Сегодня / Неделя / Месяц”
   * прогресс по активным целям (мини-индикаторы)

2. **Goals (Список целей)**

   * карточки целей с прогрессом, ближайшими задачами и рисками
   * фильтры: активные/пауза/завершённые; по областям; по приоритету

3. **Goal Detail (Страница цели)**

   * краткое описание + метрика успеха
   * **декомпозиция**: вкладки “Год / Месяцы / Недели / Дни”
   * **Timeline** (лента/Гантт-лайт) с вехами и задачами
   * журнал изменений (опционально)

4. **Calendar / Timeline (общий)**

   * календарь задач (день/неделя/месяц)
   * режим “таймлайн” для целей и вех

5. **Reviews**

   * недельный обзор (план/факт, переносы)
   * месячный обзор (корректировка целей/планов)

6. **Settings**

   * неделя начинается с (Mon/Sun), часовой пояс, нотификации, интеграции (в перспективе)

## 5) Основные сценарии и требования

### 5.1 Создание цели за 1–2 клика (критично)

**Требования:**

* Глобальная кнопка “+” и/или командная строка (Cmd/Ctrl+K).
* Минимальная форма: **Название цели** + (опционально) “метрика”.
* После сохранения — сразу предложить “Разложить цель” (one-click wizard) или начать с 1-го шага “первое действие на этой неделе”.

**Acceptance criteria:**

* Создание цели возможно за ≤10 секунд без переходов по многоэкранным мастерам.
* Сразу после создания пользователь видит первый рекомендуемый плановый элемент (например, задача на текущую неделю).

### 5.2 Декомпозиция цели: год → месяц → неделя → день

**Требования:**

* Два режима:

  1. **Авто-план**: система предлагает структуру (по шаблону) и раскладывает по периодам.
  2. **Ручной план**: пользователь добавляет вехи/задачи по месяцам/неделям.
* Любой PlanItem может иметь дочерние элементы (например “Веха месяца” → “Недели” → “Дни”).
* Быстрое “свернуть/развернуть” уровни.

**Правила разложения:**

* “Месячные” элементы автоматически разбиваются на недели по выбранному календарю.
* “Недельные” могут автоматически порождать daily-задачи при включении режима “ежедневная рутина”.
* Для повторяющихся задач поддержка: ежедневно/по будням/по конкретным дням недели/раз в N недель/ежемесячно.

### 5.3 Визуализация прогресса и контроль

**Требования:**

* Для каждой цели:

  * прогресс-бар (по выполненным задачам) и/или по метрике (current/target)
  * “следующая важная задача” и “просрочено/в риске”
* Для периодов:

  * недельная панель: план/факт (кол-во задач, effort)
  * месячный итог: выполнено/пропущено/перенесено

**Таймлайн:**

* Мини-Гантт: вехи как точки, задачи как отрезки, повторяющиеся — паттерном.
* Drag-and-drop перенос задач между днями/неделями/месяцами (с пересчётом связей).

### 5.4 Быстрое добавление задач (1–2 клика)

**Требования:**

* Быстрый инпут в “Today”: текст + горячие модификаторы:

  * #goal - автоматическая привязка к цели 
  * `tomorrow/next week/15 Jan` дата
  * `every Mon` повтор
  * `!` приоритет
* Если цель не указана — система спрашивает одним кликом: “Привязать к цели?” (показывает последние 3 цели).

### 5.5 Редактирование, перенос, завершение

**Требования:**

* “Done/Undone” одним кликом из любого списка.
* “Skip” (пропустить) для повторяющихся задач (не ломая серию).
* Быстрый перенос: “на завтра / на следующую неделю / в бэклог”.
* Массовые операции на списке (multi-select): перенести/удалить/назначить цель.

### 5.6 Обзоры (ритуалы) как механизм удержания

**Недельный обзор (обязательная фича для качества):**

* Автогенерация по воскресеньям/понедельникам:

  * что сделано
  * что просрочено
  * что перенести
  * “1 ключевой фокус недели”
* Итог обзора формирует план недели.

**Месячный обзор:**
* сверка прогресса по целям
* корректировка вех и метрик
* предложение “перепланировать” при отставании

---

## 6) Права, состояния, логика статусов

**Статусы цели:**

* Active: участвует в “Today/Week”
* Paused: не генерирует задачи, но хранится
* Done: фиксируется дата завершения, элементы закрываются/архивируются
* Archived: скрыта из основных экранов

**Статусы задач:**

* Todo
* Done
* Skipped (отдельно от Done)
* Failed (опционально; чаще достаточно skipped)

---

## 9) MVP-срез (что обязательно, чтобы продукт “заработал”)

1. Создание цели (минимальная форма)
2. Быстрое добавление задач + привязка к цели
3. Декомпозиция вручную на месяц/неделю/день (без сложного мастера)
4. Today + Week списки
5. Повторяющиеся задачи (минимум: weekly и daily)
6. Done/Undone/перенос на другую дату
7. Прогресс по цели (простая модель: % выполненных задач / метрика)
8. Недельный обзор (упрощённый: перенос просроченного и выбор фокуса недели)
